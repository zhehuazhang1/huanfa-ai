# 上线配置清单

这份清单给正式部署和真实 POC 使用。当前本地代码已经可以用 Mock 模式跑通顾客、商家、平台、租户隔离；上线前要把下面这些 Mock 配置逐步换成真实服务。

## 1. 环境变量分层

本地演示：

```text
APP_ENV=local
PAYMENT_PROVIDER=mock
TEMP_STORAGE_PROVIDER=mock
FEISHU_SYNC_PROVIDER=mock
DIFY_BASE_URL=
DIFY_API_KEY=
```

真实 POC：

```text
APP_ENV=staging
PAYMENT_PROVIDER=mock
TEMP_STORAGE_PROVIDER=aliyun_oss
FEISHU_SYNC_PROVIDER=feishu_webhook
DIFY_BASE_URL=https://你的dify域名
DIFY_API_KEY=你的Dify工作流Key
```

正式商业 MVP：

```text
APP_ENV=production
DATABASE_URL=mysql+pymysql://账号:密码@mysql:3306/hair_ai
REDIS_URL=redis://redis:6379/0
PAYMENT_PROVIDER=wechat_pay
TEMP_STORAGE_PROVIDER=aliyun_oss
FEISHU_SYNC_PROVIDER=feishu_webhook
DIFY_BASE_URL=https://你的dify域名
DIFY_API_KEY=你的Dify工作流Key
```

## 2. 必填密钥

平台统一持有以下密钥，第一版不让客户自己填 API Key：

| 服务 | 用途 | 本地演示 | 真实 POC | 正式上线 |
|---|---|---:|---:|---:|
| Dify | 调用换发型发色工作流 | 可空 | 必填 | 必填 |
| 通义万相/DashScope | Dify 内部图像编辑模型 | 可空 | 必填 | 必填 |
| 阿里云 OSS | 自拍和生成图短期存储 | 可空 | 必填 | 必填 |
| 微信小程序 | 顾客/商家登录 | 可模拟 | 建议接入 | 必填 |
| 微信支付 | AI试发付费 | Mock | 可暂缓 | 必填 |
| 飞书 | 统计同步 | Mock | 建议接入 | 建议接入 |

## 3. Dify 工作流必须满足

输入必须接收：

```json
{
  "job_no": "AI202605270001",
  "direction": "female",
  "selected_style": {},
  "selected_color": {},
  "recommendations": [],
  "photo_temp_url": "https://short-lived-photo-url"
}
```

输出必须返回：

```json
{
  "status": "success",
  "internal_api_cost": 0.88,
  "images": [
    {"slot": "main", "temp_image_url": "https://short-lived-result-main.jpg"},
    {"slot": "natural", "temp_image_url": "https://short-lived-result-natural.jpg"},
    {"slot": "advanced", "temp_image_url": "https://short-lived-result-advanced.jpg"}
  ]
}
```

硬规则：

1. 成功必须返回 3 张图。
2. 主图是顾客选择的发型/发色。
3. 另外 2 张是推荐方案。
4. 必须基于用户原自拍换发型发色。
5. 尽量只改头发，不改五官、脸型、服装、背景。
6. 少于 3 张图、超时、失败都不扣客户 AI 次数。
7. `internal_api_cost` 只给平台后台看，不能给客户老板端和顾客端看。

## 4. OSS 存储规则

正式环境必须使用短 TTL：

```text
TEMP_STORAGE_PROVIDER=aliyun_oss
OSS_BUCKET=你的bucket
OSS_REGION=oss-cn-hangzhou
OSS_ACCESS_KEY_ID=你的AccessKey
OSS_ACCESS_KEY_SECRET=你的Secret
```

规则：

1. 自拍只用于本次生成。
2. 生成图只临时展示。
3. 用户自行长按保存或截图。
4. 平台不长期保存顾客自拍和生成图。
5. object key 必须包含 `tenant_id/store_id/user_id`，方便隔离和排查。
6. 后端在 AI 调用结束后主动删除自拍；OSS 仍需为 `temp/` 路径配置生命周期自动删除规则，作为清理失败时的兜底。
7. 微信公众平台需要把 OSS Bucket 的 HTTPS 域名配置为 request 合法域名，供小程序 PUT 上传自拍。

## 5. 微信支付规则

服务订单仍然到店支付。

AI试发付费是独立流程：

1. 店内扫码免费 2 次。
2. 店员可赠送 1 次，自动记录赠送人和转化。
3. 不在店顾客也可以用，但要付费。
4. 付费成功但生成失败，允许免费重试一次。
5. 正式商业 MVP 必须接微信支付回调验签。

## 6. 上线前必须跑的验收

```powershell
$env:PYTHONPATH='backend'
python -m pytest backend/tests -q
python backend/scripts/acceptance_check.py
```

还要检查：

```bash
curl https://api.yourdomain.com/health
curl https://api.yourdomain.com/platform/deployment-readiness
```

生产环境 `deployment-readiness` 不能出现以下 blocker：

1. Dify 仍是 mock。
2. OSS 仍是 mock。
3. 微信支付仍是 mock。

## 7. 真实 POC 验收标准

先不要急着卖正式版本，先用真实 Dify + 通义万相跑 POC：

1. 男性、女性、中性各准备至少 5 张自拍。
2. 分别测试只换发型、只换发色、发型+发色。
3. 平均生成时间控制在 45 秒以内。
4. 记录每次真实 API 成本。
5. 记录是否像本人、是否只改头发、是否保留服装背景。
6. 可展示率达到客户能接受的水平，再进入正式 MVP。

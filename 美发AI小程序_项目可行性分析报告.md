# 美发AI小程序项目可行性分析报告

版本：v1.0  
日期：2026-05-25  
适用规模：15家门店，每家门店每月约1000名顾客，总计约15000名顾客/月  
报告目的：判断项目是否值得做、哪些功能先做、哪些功能有风险、整体业务逻辑是否跑得通

---

## 1. 结论摘要

本项目整体具备可行性，适合分阶段落地。

### 1.1 客户确认后的核心体验修正

本项目的客户展示和第一阶段MVP，核心体验应修正为：

1. 顾客先选择自己想要的发型性别风格：男性、女性、中性。
2. 顾客选择目标发色。
3. 顾客选择目标发型。
4. 顾客拍照或上传照片。
5. AI同步生成顾客本人换发效果。
6. 系统同时推荐2个备选发型。
7. 系统同时推荐擅长该风格、该发色、该发型的发型师。
8. 顾客满意后进入预约。

因此，第一阶段不是以“纯问答推荐”为主，而是以“自选发型发色 + AI同步换发预览 + 备选推荐 + 发型师匹配”为主。问答式推荐可以作为辅助入口，而不是主入口。

从业务角度看，美发门店存在明显的数字化需求：顾客选发型困难、预约沟通成本高、发型师匹配依赖人工、门店经营数据分散、会员复购缺少自动化工具。AI小程序可以帮助门店提升咨询效率、预约转化、会员复购和老板管理效率。

从技术角度看，微信小程序、后端系统、数据库、AI推荐、门店看板、会员体系均属于成熟技术范围，可以落地。项目难点不在于“能不能做”，而在于第一阶段不能贪大，必须先把预约、发型发色推荐、门店权限、数据隔离这几件事做稳。

建议项目采用“三阶段”推进：

1. 第一阶段：展示小样 + MVP核心系统。
2. 第二阶段：会员运营、作品集、飞书后台、AI数据分析。
3. 第三阶段：虚拟换发、库存、抖音/美团/大众点评、支付闭环。

## 2. 项目整体可行性

### 2.1 市场和业务可行性

可行。

美发门店的典型痛点包括：

1. 顾客不知道自己适合什么发型。
2. 顾客咨询时间长，前台和发型师沟通成本高。
3. 男客、女客需求差异大，传统图册容易混乱。
4. 发型师擅长方向不同，人工匹配效率低。
5. 多门店老板难以及时看到每家店经营情况。
6. 会员生日、流失召回、复购提醒主要靠人工。

AI小程序可以直接解决其中一部分问题：

1. 通过问答推荐合适发型和发色。
2. 通过规则过滤区分男客、女客和通用发型。
3. 通过预约系统减少人工沟通。
4. 通过门店看板让老板掌握15家店数据。
5. 通过会员运营提高复购。

业务上具备落地价值。

### 2.2 客户规模可行性

可行，但要按15家门店设计，不能按单店小程序设计。

当前规模：

1. 门店数量：15家。
2. 每店每月顾客：约1000人。
3. 总顾客量：约15000人/月。
4. 日均顾客量：约500人/天。

这个体量不算超大，但已不是简单展示型小程序。系统必须具备：

1. 门店数据隔离。
2. 多角色权限。
3. 高峰期稳定性。
4. 数据统计能力。
5. AI接口失败时的降级方案。

只要架构从一开始按多门店设计，15家门店规模可以承载。

### 2.3 技术可行性

可行。

项目涉及的主要技术均成熟：

1. 微信小程序：适合顾客预约、会员中心、商家轻后台。
2. FastAPI后端：适合接口开发、权限管理、业务逻辑。
3. MySQL数据库：适合门店、会员、预约、支付、统计数据。
4. Redis缓存：适合缓存AI推荐结果和会话。
5. Dify或自建AI流程：适合发型推荐、客服问答、数据分析。
6. OSS对象存储：适合保存发型图、作品集图片。
7. 微信订阅消息：适合预约提醒、生日提醒、召回提醒。

技术上不存在不可实现的问题。

### 2.4 资金和周期可行性

可行，但必须控制第一阶段范围。

如果第一阶段同时做预约、AI推荐、虚拟换发、库存、支付、飞书、抖音、美团、点评、会员营销，会导致周期变长、成本上升、风险增大。

更合理的方式是：

1. 先做客户展示小样，用于确认方向和签约。
2. 再做MVP，把核心业务跑通。
3. 上线1家门店灰度。
4. 再推广到15家门店。
5. 根据实际使用情况增加高级功能。

这样资金投入更可控，也更容易成功。

## 3. 功能模块可行性分析

### 3.1 微信登录和会员身份

可行性：高  
建议优先级：第一阶段必须做

功能说明：

1. 顾客用微信进入小程序。
2. 系统识别顾客身份。
3. 顾客可以查看自己的预约、积分、优惠券。

可行原因：

1. 微信小程序登录能力成熟。
2. 通过 openid 可以稳定识别用户。
3. 后端可以用 JWT 管理登录状态。

风险点：

1. 手机号需要用户授权，不能默认获取。
2. 用户隐私政策必须完善。

结论：

该模块成熟可靠，必须作为基础能力优先完成。

### 3.2 门店选择和15店数据隔离

可行性：高  
建议优先级：第一阶段必须做

功能说明：

1. 顾客进入后选择门店。
2. 店长只能看本店数据。
3. 老板可以看全部15家门店。
4. 发型师只能看自己的预约和业绩。

可行原因：

1. 数据库加 `store_id` 即可实现门店隔离。
2. 后端通过角色权限控制访问范围。

风险点：

1. 不能只靠前端隐藏数据。
2. 所有接口都必须在后端强制校验权限。

结论：

技术可行，且是15店系统的核心底座。必须第一阶段完成。

### 3.3 发型库

可行性：高  
建议优先级：第一阶段必须做

功能说明：

1. 顾客可以浏览发型。
2. 发型按男、女、通用区分。
3. 发型按短发、中发、长发区分。
4. 发型按脸型、风格、服务类型打标签。

可行原因：

1. 本质是结构化数据管理和图片展示。
2. 难度较低，适合作为第一阶段核心模块。

风险点：

1. 如果标签不规范，AI推荐会不准确。
2. 发型素材需要客户或门店提供，或者先用示例素材。

结论：

完全可行。必须明确男客发型、女客发型、通用发型，不能混在一起。

### 3.4 发色库

可行性：高  
建议优先级：第一阶段建议做

功能说明：

1. 发色按男、女、通用区分。
2. 发色按自然色、棕色、亚麻色、红色、潮色等分类。
3. 标记是否需要漂发。
4. 标记是否职场友好。

可行原因：

1. 发色也是结构化数据和图片展示。
2. 可以和AI推荐直接联动。

风险点：

1. 发色效果与真实发质、底色有关，不能承诺100%一致。
2. 漂发、潮色需要在推荐中明确提醒风险和护理成本。

结论：

可行，建议第一阶段做基础版。尤其要区分男士常见自然发色和女士潮色。

### 3.5 AI同步换发预览和推荐

可行性：中高  
建议优先级：第一阶段必须做

功能说明：

顾客操作：

1. 选择想要的发型性别风格：男性、女性、中性。
2. 选择目标发色。
3. 选择目标发型。
4. 拍照或上传照片。

系统返回：

1. AI换发预览效果图。
2. 2个备选推荐发型。
3. 推荐理由。
4. 擅长该发型、发色、风格的发型师。

可行原因：

1. 原方案已经包含发型预览、style_id、color_id、Face++或同类AI换发能力。
2. 发型和发色可以结构化管理，顾客选择后再生成。
3. 备选推荐可以通过规则过滤产生，AI负责解释和排序。
4. 发型师推荐可以根据门店、技能标签、服务风格和作品标签匹配。

关键逻辑：

1. 顾客先主动选择目标风格、发色、发型。
2. 拍照前必须弹隐私授权。
3. 照片只用于本次AI生成，不保存到数据库。
4. AI生成预览图后，系统同步推荐2个备选发型。
5. 系统同步推荐擅长该风格的发型师。
6. 如果AI生成失败，仍保留顾客已选方案，并返回备选发型和发型师。

风险点：

1. AI换发涉及照片和人脸隐私，合规要求高。
2. 预览效果和真实烫染效果可能存在差异。
3. 不能把Face++或其他AI服务长期密钥放在小程序前端。
4. 不能保存顾客照片。
5. 不能让AI编造数据库不存在的发型或发型师。

结论：

可行，且应作为第一阶段核心展示能力。但必须采用“顾客自选 + AI生成 + 规则备选推荐 + 发型师匹配 + 隐私保护”的方式。

### 3.6 发型师匹配

可行性：高  
建议优先级：第一阶段必须做

功能说明：

系统根据以下条件推荐发型师：

1. 当前门店。
2. 服务性别。
3. 擅长项目。
4. 可预约时间。
5. 评分和历史作品。

可行原因：

1. 本质是规则匹配和排序。
2. 不依赖复杂AI即可完成。

风险点：

1. 发型师技能标签必须录入准确。
2. 预约时间冲突必须严格校验。

结论：

可行，且非常适合作为客户展示亮点。

### 3.7 预约系统

可行性：高  
建议优先级：第一阶段必须做

功能说明：

1. 顾客选择门店、发型师、套餐、时间。
2. 系统生成预约。
3. 店长或前台确认预约。
4. 服务完成后更新会员数据。

可行原因：

1. 预约系统是标准业务模块。
2. 与门店、发型师、会员数据天然关联。

风险点：

1. 同一发型师同一时间不能重复预约。
2. 取消、改约、迟到、未到店规则要后续明确。

结论：

完全可行，是项目第一阶段的核心闭环。

### 3.8 商家后台

可行性：高  
建议优先级：第一阶段做基础版

功能说明：

1. 店长查看本店预约。
2. 店长管理套餐、发型师。
3. 店长查看本店数据。
4. 老板查看全部门店数据。

可行原因：

1. 小程序内可以做轻后台。
2. 后端权限控制即可支持不同角色。

风险点：

1. 小程序后台不适合承载太复杂的管理操作。
2. 大量数据录入后续可以交给飞书或网页后台。

结论：

可行。第一阶段建议做轻后台，满足预约管理和基础数据查看。

### 3.9 老板15店数据看板

可行性：高  
建议优先级：第一阶段建议做基础版

功能说明：

老板可以看到：

1. 15家门店预约数量。
2. 到店完成数量。
3. 营收数据。
4. 客单价。
5. 新老客比例。
6. 门店排行。
7. 发型师排行。

可行原因：

1. 数据来自预约、服务记录、支付记录。
2. 可以用每日汇总表提高查询效率。

风险点：

1. 如果没有支付记录，只能统计预约和到店，营收会不准确。
2. 看板数据口径必须提前定义。

结论：

可行。第一阶段先做基础看板，后续再做AI解读。

### 3.10 会员积分和优惠券

可行性：高  
建议优先级：第一阶段可做基础版，第二阶段增强

功能说明：

1. 消费后增加积分。
2. 会员等级升级。
3. 发放优惠券。
4. 生日优惠。
5. 流失召回。

可行原因：

1. 积分和优惠券是成熟业务逻辑。
2. 与复购提升直接相关。

风险点：

1. 优惠券规则不能太复杂。
2. 不能在第一阶段做储值卡，储值涉及预付款合规风险。

结论：

积分和优惠券可行；充值储值第一阶段不建议做。

### 3.11 微信订阅消息

可行性：中  
建议优先级：第二阶段

功能说明：

1. 预约提醒。
2. 生日提醒。
3. 召回提醒。
4. 活动提醒。

可行原因：

1. 微信官方支持订阅消息。
2. 适合服务提醒。

风险点：

1. 不能强制推送，必须用户主动授权。
2. 模板消息类目需要申请。
3. 不能把它当短信群发工具。

结论：

可行，但受微信规则限制。第一阶段可预留，第二阶段重点做。

### 3.12 虚拟换发

可行性：中  
建议优先级：第三阶段

功能说明：

顾客上传照片，系统生成换发型、换发色效果图。

可行原因：

1. 市面上已有第三方AI换发服务。
2. 技术上可以通过API或SDK接入。

风险点：

1. 顾客照片、人脸属于敏感数据，合规要求高。
2. 第三方服务成本和效果不稳定。
3. 照片不能保存到数据库。
4. API密钥不能放在小程序前端。
5. 生成效果可能与真实烫染结果有差异。

结论：

可行，但不建议作为第一阶段核心功能。可以在客户展示中作为未来亮点，不要承诺第一版必须上线。

### 3.13 发型师作品集

可行性：高  
建议优先级：第二阶段

功能说明：

1. 发型师上传作品。
2. 顾客浏览作品。
3. 作品按烫发、染发、剪发、男士、女士分类。

可行原因：

1. 本质是图片上传和标签管理。
2. 对顾客选择发型师有帮助。

风险点：

1. 涉及顾客真人照片时，需要获得授权。
2. 图片需压缩和审核。

结论：

可行，建议第二阶段上线。

### 3.14 飞书多维表格后台

可行性：中高  
建议优先级：第二阶段

功能说明：

使用飞书做数据录入、表格管理、简单看板。

可行原因：

1. 飞书适合非技术人员录入和管理。
2. 可以降低后台开发成本。

风险点：

1. 飞书和MySQL必须做好同步。
2. 数据源必须明确，不能两边都能随意改。

结论：

可行，但建议系统核心数据仍以MySQL为主，飞书作为辅助后台。

### 3.15 产品零售和库存

可行性：中高  
建议优先级：第三阶段

功能说明：

1. 管理洗护产品库存。
2. 销售后自动扣库存。
3. 低库存提醒。

可行原因：

1. 标准库存逻辑可以实现。
2. 对连锁门店有管理价值。

风险点：

1. 门店实际库存盘点可能不准。
2. 如果没有严格销售录入，系统库存会和实际库存不一致。

结论：

可行，但需要门店配合执行。建议第三阶段做。

### 3.16 抖音、美团、大众点评接口

可行性：中  
建议优先级：第三阶段

功能说明：

同步团购订单、消费数据、平台业绩。

可行原因：

1. 平台如果开放接口，可以对接。
2. 对老板看统一经营数据有价值。

风险点：

1. 平台接口申请不一定顺利。
2. 不同平台规则和权限不同。
3. 对接周期不可完全由本项目控制。

结论：

可以预留接口，但不要把它作为第一阶段交付承诺。

### 3.17 微信支付和储值

可行性：支付可行，储值需谨慎  
建议优先级：第三阶段

功能说明：

1. 微信支付预约定金或到店支付。
2. 储值卡、余额充值。

可行原因：

1. 微信支付技术成熟。

风险点：

1. 储值卡涉及预付款和法律合规。
2. 退款、账务、对账、发票、纠纷处理都需要规则。

结论：

微信支付可做；储值卡第一阶段不建议做，必须客户确认资质和法务风险后再做。

## 4. 业务逻辑可行性分析

### 4.1 顾客使用逻辑

可行。

推荐流程：

1. 顾客进入小程序。
2. 选择门店。
3. 选择AI推荐或手动浏览。
4. 回答性别、脸型、发长、风格、烫染接受度。
5. 系统先按规则筛选发型发色。
6. AI生成TOP3推荐和理由。
7. 顾客选择一个方案。
8. 系统推荐合适发型师。
9. 顾客选择时间并提交预约。
10. 店长确认。
11. 到店服务。
12. 服务完成后进入会员沉淀。

该流程符合真实美发消费路径。

### 4.2 男女顾客区分逻辑

可行，且必须做成硬规则。

正确逻辑：

1. 顾客选择男客，系统只展示 `male` 和 `unisex` 发型发色。
2. 顾客选择女客，系统只展示 `female` 和 `unisex` 发型发色。
3. 顾客选择不限，系统展示 `unisex` 或要求进一步选择。
4. AI不得突破这个限制。

为什么不能只靠AI提示词：

1. AI可能理解偏差。
2. AI可能编造结果。
3. AI可能推荐数据库不存在的发型。

所以必须由后端程序先过滤，再让AI做解释。

### 4.3 发型发色推荐逻辑

可行。

推荐规则应包含：

1. 性别。
2. 脸型。
3. 当前发长。
4. 目标风格。
5. 是否接受烫发。
6. 是否接受染发。
7. 是否接受漂发。
8. 是否职场友好。
9. 年龄段。

逻辑可行点：

1. 发型和发色可以用标签管理。
2. 后端规则可以保证底线正确。
3. AI可以让推荐话术更像真人顾问。

风险：

1. 发型标签越粗糙，推荐越普通。
2. 真实效果受发质、脸型细节、发量影响。

建议：

第一版不承诺“绝对适合”，话术应为“推荐参考，最终以发型师面诊为准”。

### 4.4 预约逻辑

可行。

预约必须处理：

1. 门店。
2. 顾客。
3. 发型师。
4. 套餐。
5. 发型。
6. 发色。
7. 预约时间。
8. 预约状态。

状态建议：

```text
pending     待确认
confirmed   已确认
done        已完成
cancelled   已取消
```

风险：

1. 发型师时间冲突。
2. 顾客临时取消。
3. 店长未及时确认。

建议：

第一版先做简单预约和确认，后续再做复杂排班。

### 4.5 门店权限逻辑

可行。

正确权限：

1. 老板看全部门店。
2. 店长看本店。
3. 发型师看本人。
4. 顾客看本人。

该逻辑清晰，技术实现成熟。

最大风险：

后端如果没有强制过滤，可能发生跨店数据泄露。

解决方式：

每个接口都根据登录用户角色自动追加查询条件，不信任前端传来的 `store_id`。

### 4.6 老板看板逻辑

可行。

看板数据来源：

1. 预约表。
2. 服务完成记录。
3. 支付记录。
4. 会员表。
5. 每日汇总表。

第一阶段建议展示：

1. 今日预约数。
2. 本月预约数。
3. 本月到店数。
4. 本月营收。
5. 客单价。
6. 门店排行。

风险：

如果不做支付或服务完成录入，营收只能估算。

建议：

第一阶段可先做预约和到店数据，营收可后续接支付或人工录入。

### 4.7 会员复购逻辑

可行。

会员运营路径：

1. 顾客预约后成为会员。
2. 完成服务后累计积分。
3. 根据消费金额和到店次数升级等级。
4. 生日发优惠券。
5. 超过60天未到店发召回提醒。

风险：

微信订阅消息不能强制发，必须用户授权。

建议：

第一阶段先记录会员和积分，第二阶段再做自动推送。

## 5. 风险评估

### 5.1 高风险事项

1. 虚拟换发涉及顾客照片和人脸隐私。
2. 储值卡涉及预付款合规。
3. 外部平台接口受平台审核影响。
4. AI如果没有规则约束，可能推荐错误。
5. 多门店权限如果做错，会造成数据泄露。

### 5.2 中风险事项

1. 发型标签不规范导致推荐不准。
2. 门店员工不配合录入数据，导致看板不准。
3. 微信订阅消息授权率不高。
4. 高峰期AI接口超时。
5. 图片素材质量影响展示效果。

### 5.3 低风险事项

1. 微信登录。
2. 发型库展示。
3. 套餐展示。
4. 基础预约。
5. 基础看板。

## 8. 最终判断

本项目值得做，且具备落地可行性。

最稳妥的路径是：

1. 先用展示小样打动客户。
2. 第一阶段只做核心闭环。
3. 强制落实男女性别区分和门店权限隔离。
4. AI推荐采用“规则过滤 + AI解释 + 后端校验 + 失败兜底”。
5. 先灰度1家门店，再逐步推广到15家门店。
6. 虚拟换发、储值卡、外部平台接口不要放进第一阶段硬承诺。

如果按这个路径执行，项目可控、可上线、可运营，也便于后续扩展。
